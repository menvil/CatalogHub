# CatalogHub v0.2.1 — Phase 21 Production Readiness

Release date: 2026-07-16  
Target: controlled production / public demo

## Summary

Phase 21 stabilizes CatalogHub for its first controlled launch. It adds explicit readiness evidence, baseline security and health diagnostics, critical-flow smoke tests, operational audits, and actionable deployment/recovery documentation. No new product bounded context or distributed synchronization architecture was introduced.

## Readiness and Security

- Added a repository-owned production readiness checklist and environment audit.
- Added a secret-free `.env.production.example` with production-safe defaults.
- Added security response headers and per-IP limits for public high-frequency forms/endpoints.
- Documented and configured secure production session expectations.
- Audited the Central Admin, Site Admin, and public authorization boundaries; added explicit resource, route, and site-query guards for critical gaps.

## Health and Observability

- Added coarse public application health plus internal queue, storage, and external-source checks.
- Health checks are safe: they expose no credentials, do not run imports/syncs, and do not call live external APIs.
- Added an observability baseline with structured context for snapshot generation.
- Added a provider-neutral error-reporting configuration and data-handling policy. Reporting stays disabled until a provider, DSN, SDK, and scrubbing test are approved.

## QA and Performance

- Added fast local smoke coverage for import, projections, public pages, and admin navigation/authorization.
- Added a public-page performance budget and a database index audit.
- Added safe indexes for the verified Phase 21 query patterns.
- Added a full Phase 0–21 regression checklist with automated results, manual staging checks, known limitations, and sign-off fields.

## Operations

- Added backup/restore dry-run, deployment, and rollback runbooks.
- Runbooks include concrete preflight, migration, queue, storage, health, smoke, rollback, and incident-record steps.
- Destructive automatic restore remains intentionally out of scope.

## Known Limitations

- Manual staging checks and release-owner sign-off remain environment-specific and must be recorded during deployment.
- External error reporting is disabled until a provider and privacy/scrubbing validation are approved.
- There is no distributed tracing, infrastructure worker heartbeat, automated production restore, multi-region deployment, or full compliance program.
- Catalog snapshots are portable exports; they do not replace database and object-storage backups.
- PostgreSQL query-plan evidence is still required before adding specialized JSON facet or full-text indexes.
- CatalogHub currently uses one Filament panel; authorization depends on tested resource/page/route guards and site query scoping.

## Next Recommended Work

Perform a controlled staging deployment using the readiness checklist and runbooks, attach target-environment evidence, complete the backup restore dry run, measure representative pages against the performance budget, and record the go/no-go decision. Any later product phase should start only after the launch observation window and known-risk review.
